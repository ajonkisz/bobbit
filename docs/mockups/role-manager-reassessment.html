#1 Missing cursor: pointer on clickable rows high

The .role-row has an @click handler in the template that navigates to the edit view, and a :hover style with background change — but the CSS never sets cursor: pointer. This breaks the fundamental affordance that tells users the row is clickable. The hover background change alone is insufficient because it only signals "interactive" after the cursor is already over the element; the pointer cursor signals it before the user commits.

Bug: .role-row in role-manager.css line 117 — no cursor: pointer. The individual .role-row-action-btn does have it (line 176), creating an inconsistency where the buttons feel clickable but the row itself doesn't.

#2 No keyboard focus ring on role rows high

Role rows are <div> elements with click handlers but no tabindex, no role="button", and no :focus-visible styles. Keyboard users cannot tab to or activate role rows. This is an accessibility failure — the entire primary interaction (navigating to edit) is mouse-only.

#4 List gap too tight (2px) — rows merge visually medium

.roles-list { gap: 2px; } creates a near-continuous surface where individual rows are hard to distinguish. The 1px borders blend together. The original intent of "tighter spacing" overcorrected — rows need enough breathing room to be scannable as discrete items. 6px gap maintains the compact feel while giving each row clear separation.

#5 Mobile slug hidden entirely — loses context medium

The @media (max-width: 768px) block sets .role-row-slug { display: none; }. On mobile, users only see the display label (e.g. "Documentation Writer") with no way to distinguish roles that might have similar names. The slug is valuable context — it's the machine identifier used in team config. Instead of hiding it completely, truncate it to fit.

#6 Action buttons always visible — visual noise on desktop low

Edit and delete buttons are always visible on every row. With 5+ roles, the repeated icon pairs add visual clutter. Since the entire row is clickable (navigates to edit), the edit button is redundant at rest. Hiding actions until hover reduces noise while keeping them instantly accessible. On mobile/touch, actions must remain always-visible since there's no hover.

Trade-off: Hover-reveal actions are less discoverable for new users. However, the row-level click already handles the primary action (edit), and the delete action is destructive enough that requiring an explicit hover isn't a usability cost — it's a feature. Add @media (hover: none) fallback to keep actions visible on touch devices.

#7 Empty state bobbit too faint (opacity 0.3) low

The branded empty state is a good pattern, but opacity: 0.3 on the bobbit sprite makes it nearly invisible against the dark background. #8 Readonly field background uses color-mix at 50% — too transparent nit

The readonly "Name" field in the edit view uses background: color-mix(in oklch, var(--secondary) 50%, transparent). This makes it nearly invisible against the page background, barely distinguishable from a missing field. A full var(--secondary) background communicates "this field exists but isn't editable" more clearly, matching the disabled input convention used elsewhere in the app.

#9 Edit view breadcrumb label doesn't truncate on narrow screens low

The breadcrumb in edit view renders as Roles / {role.label} with the label in an <h1> that has text-overflow: ellipsis. But the parent .roles-title-group uses display: flex with min-width: 0 — the h1 won't truncate unless it also has min-width: 0 and overflow: hidden is inherited properly through the flex chain. On very narrow viewports (or long role labels like "Senior Full-Stack Documentation Writer"), the breadcrumb can overflow the nav bar.

What's Working Well

The recent changes addressed the major structural issues. These are solid and should not be touched:

  • List layout with bare sprites — cleaner than the old card grid; appropriate information density for a settings page
  • Slug below label — useful context without taking too much space; mono font differentiates it clearly
  • Hash routing with browser back — #/roles and #/roles/:name is clean; popstate works correctly
  • Breadcrumb nav in edit view — Roles / {label} communicates hierarchy without wasting space
  • oklch color-mix migration — modern, perceptually uniform; hover borders and destructive backgrounds look good
  • Edit view two-column layout — identity/prompt/tools on left, accessory/actions on right is logical grouping
  • Branded empty state — bobbit sprite + clear CTA is friendly and informative
  • Responsive column collapse — edit sidebar stacks below main content on mobile; correct breakpoint

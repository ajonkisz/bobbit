Role Manager — UX Improvement Proposals

Six targeted improvements to the list view. Each section shows the current implementation side-by-side with the proposed change, with rationale for every modification. All proposals use the same dark-theme HSL variables and pixel-art Bobbit sprite system as the production code. Hover over rows to see interactive states.

1. Tighter Row Spacing

Reduce vertical gap between rows and inner padding for a denser, scannable list
Problem: Current gap: 6px between rows and padding: 12px 18px per row creates excessive vertical space. With 6+ roles the list feels sparse and requires scrolling unnecessarily.
Change: Reduce gap to 2px, padding to 10px 16px, border-radius to 8px (from 10px). Removes the inner avatar box (background + border) — the Bobbit sprite sits directly in the row, saving ~8px horizontal space per row.
Current — gap: 6px, padding: 12px 18px
Proposed — gap: 2px, padding: 10px 16px

2. Remove Avatar Container Box, Add Machine Name

Drop the bordered square around the Bobbit sprite; show the role slug as secondary info
Problem: The 36×36px bordered avatar box (.role-row-avatar) adds visual weight that competes with the role label. The Bobbit sprites are already distinctive enough without a container. Also, the list shows only the display label — the machine name (slug) is hidden until you enter edit view.
Change: Remove the bordered box. Render the sprite directly at its natural size (~28×19px at 1.6× scale). Add a muted monospace slug after the label, visible at desktop widths, hidden on mobile. This gives at-a-glance info for power users who reference roles by machine name in configs.
Current — bordered avatar box
Proposed — bare sprite + slug

3. Improved Empty State

Replace the generic icon with a greyed-out Bobbit and friendlier copy
Problem: The current empty state uses a generic Users Lucide icon at low opacity. It's functional but doesn't match the Bobbit branding. The copy "No roles defined yet" is passive.
Change: Use a greyed-out Bobbit sprite as the empty-state icon (stays on brand). Add a brief description of what roles do. Tighten spacing. Same CTA button.

4. Delete Button — Reveal on Hover

Reduce accidental deletions; keep the row visually clean at rest
Problem: The delete button is always visible with the same weight as the edit button. This creates a "dangerous" click target that's always present. The two icon buttons side-by-side have equal prominence — nothing signals which is safe vs destructive until hover.
Change: Hide the delete button at rest (opacity: 0), reveal on row hover. This keeps the resting state clean (only the edit pencil visible) and prevents accidental clicks. On mobile (no hover), both buttons stay visible since touch targets need to be always accessible.
Current — both always visible
Proposed — delete hidden until hover (try it)

5. Mobile Layout (320px viewport)

Compact rows with smaller touch targets, both action buttons always visible
Problem: Current responsive breakpoint at 768px reduces padding slightly but doesn't adjust button sizes or hide the slug. Rows still feel desktop-proportioned on small screens.
Change: At mobile widths: hide the monospace slug (label only), reduce icon buttons to 28px, reduce row gap to 10px, keep both edit + delete visible (no hover on touch). The "New Role" button in the nav uses a compact icon-only variant.

6. List Width & Centering

Center the list horizontally for wide viewports, increase max-width slightly
Problem: Current max-width: 560px with margin: 0 auto is slightly narrow — long role labels like "Documentation Writer" plus the slug plus two buttons feel cramped. Also the list is left-aligned (no margin: 0 auto was set on .roles-body directly, but the list inside has it).
Change: Widen to max-width: 600px. Keep margin: 0 auto. Subtle change but removes label truncation for typical role names.

Design Rationale

  • Density: Tighter gaps (2px) and lighter padding (10px 16px) make the list feel purposeful, not padded. Six roles fit comfortably without scrolling.
  • Visual hierarchy: Removing the avatar border box reduces noise. The Bobbit sprite is already a strong visual anchor — it doesn't need a container to be readable.
  • Information density: The monospace slug gives power users the machine name at a glance, avoiding a round-trip to the edit view.
  • Safety: Hiding delete on hover follows the principle of least surprise. Destructive actions should require intentional discovery, not sit next to the primary action at equal prominence.
  • Branding: The empty state uses a Bobbit sprite instead of a generic icon, keeping the page cohesive with the rest of the app's identity.
  • Mobile: Touch targets remain accessible (28px+), both buttons always visible (no hover on touch), slug hidden to save space. The nav "New Role" becomes icon-only.
  • Constraints respected: All colours use the existing HSL custom properties. The pixel-art sprite system is identical (box-shadow at 1.6× scale with hue-rotate). No new colours, fonts, or design tokens introduced.

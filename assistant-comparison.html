Bug

Session restore loses specialized prompt after server restart Role Spec Tool
restoreSession() in session-manager.ts only has an explicit if (ps.goalAssistant) branch for reassembling the specialized system prompt. Role, tool, and spec assistant sessions fall through to the generic branch (base system prompt + AGENTS.md + goal spec), losing their specialized prompts entirely. After a server restart, these sessions behave as normal sessions instead of assistants.

Gaps (Spec Assistant)

No draft persistence Spec
Goal and role assistants auto-save all editable preview fields to IndexedDB on every keystroke, restore them on reconnect, and delete the draft when the entity is created. The spec assistant has no persistence — if the user navigates away, refreshes the page, or the server restarts, all preview edits are lost.
Session not cleaned up after "Create Spec" Spec
Goal and role assistants delete the assistant session via DELETE /api/sessions/:id, call clearSessionModel(), and refreshSessions() after creating the entity. The spec assistant disconnects but leaves the session alive in the sidebar.
No markdown preview toggle Spec
Goal has an Edit/Preview toggle for the spec field (rendered markdown vs textarea). Role has the same for the prompt field. Spec assistant's text areas (description, must-have, should-have, must-not-have) are always in edit mode.
